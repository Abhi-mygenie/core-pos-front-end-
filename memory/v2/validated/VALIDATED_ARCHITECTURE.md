### 1.1 Order Lifecycle — End-to-End

**Confidence**: HIGH — traced through all code paths

#### State Transitions

```
[No Order] → Place Order → [running/preparing] → Update Order (add items) → [running/preparing]
                                     │
                                     ├── Cancel Item → [running] (if items remain)
                                     ├── Cancel Item → [cancelled] (if last item)
                                     ├── Cancel Order → [cancelled] → REMOVED from context
```

#### Order Removal Rule (CONFIRMED)
An order is removed from `OrderContext` when:
1. `handleUpdateOrderStatus` fetches order and status is `'cancelled'` or `'paid'` → `removeOrder(orderId)` — `socketHandlers.js:297-300`

3. `handleCancelOrder` explicitly awaits `waitForOrderRemoval(orderId, 5000)` — `OrderEntry.jsx:500`

**No other code path removes orders.** The socket handler is the sole removal mechanism during normal operation.

### RV-1: Does the backend recalculate financial totals or trust frontend values?

- **Description**: Every order payload sends frontend-calculated `order_sub_total_amount`, `tax_amount`, `gst_tax`, `vat_tax`, `order_amount`, `round_up`. Unknown if backend recalculates or uses as-is.
- **Why uncertain**: No backend code available. Comment "Rounding logic matching backend" in `orderTransform.js:324` suggests they tried to match, implying backend DOES calculate independently. But this is inference.
- **How to validate**:
  1. Place an order with known items (e.g., 1x item at ₹100, 5% GST Exclusive)
  2. Deliberately modify `order_amount` in payload to a wrong value (e.g., ₹999) using browser devtools network intercept
  3. Check: Does the API return the modified ₹999 or the correct ₹105?
  4. Check: Does the order appear in reports with ₹999 or ₹105?
- **Impact if backend TRUSTS frontend**: RISK-001 (triple tax calc) becomes CRITICAL — any frontend calculation error directly corrupts financial data. All 3 calculation locations must produce identical results.
- **Impact if backend RECALCULATES**: RISK-001 severity drops to LOW (display-only issue). Frontend calculations become advisory.
- **Cross-ref**: RISK-001, RISK-007, Q-A1

### RV-2: What happens when `waitForTableEngaged` times out?

- **Description**: After order operations, code calls `await waitForTableEngaged(tableId, 5000)`. On timeout, returns `false`, but `onClose()` executes anyway.
- **Why uncertain**: Cannot determine from code if the socket event eventually arrives late, or if the table gets stuck.
- **How to validate**:
  1. Place an order on a slow network (throttle to 3G in devtools)
  2. Monitor console for `[TableContext] waitForTableEngaged: timeout for ${tableId}`
  3. After timeout + redirect: check if table shows correct status on dashboard
  4. Click the table again — is it clickable or stuck as engaged?
- **Impact if socket never arrives**: Table appears stale on dashboard. User can still click (not engaged). Order may appear on next refresh or late socket delivery.
- **Impact if socket arrives late**: Table briefly shows wrong status, then self-corrects. Cosmetic issue only.
- **Cross-ref**: Q-A2

### RULE-1: Financial Source of Truth (Refined — see also Section 8)

**Until RV-1 is validated, assume the backend TRUSTS frontend financial values.**

| Scenario | Classification | Calculated By | Sent to API? | Authoritative? |
|----------|---------------|---------------|-------------|----------------|
| Cart total before placing | **PREVIEW** | `OrderEntry.jsx` lines 331-364 | NO | No — display hint only |
| Place Order payload | **AUTHORITATIVE SUBMISSION** | `buildCartItem()` + `calcOrderTotals()` | YES — multipart/form-data | YES — assumed backend trusts |
| Update Order payload | **AUTHORITATIVE SUBMISSION** | `buildCartItem()` + `calcOrderTotals()` over ALL items | YES — JSON body | YES — assumed backend trusts |
| Dashboard display after place | **API ECHO** | `orderFinancials` from GET single-order via socket handler | NO — read-only | YES — this IS backend truth echoed back |
| Payment panel bill summary | **PREVIEW with SUBMISSION risk** | `CollectPaymentPanel` local recalculation | Indirectly — feeds `paymentData` | See RULE-1a below |
| Collect Bill payload | **AUTHORITATIVE SUBMISSION** | `CollectPaymentPanel` → `paymentData` → `collectBillExisting()` | YES — JSON body | YES — assumed backend trusts |

#### RULE-1a: The CollectPaymentPanel Problem

The Collect Bill flow has a specific architectural issue: it recalculates totals locally from cart items and sends THOSE values (not the API-echoed `orderFinancials`) to the backend.

**Current actual data flow** (confirmed line-by-line):
```
Cart items (placed items from OrderContext + unplaced from local state)
  → CollectPaymentPanel.getItemLinePrice()         [local calc — line 48-59]
  → CollectPaymentPanel.taxTotals useMemo           [local calc — line 64-86]
  → CollectPaymentPanel.handlePayment() builds paymentData  [line 192-213]
  → OrderEntry.onPaymentComplete() passes to collectBillExisting()  [line 731]
  → API receives locally-calculated values
```

The `orderFinancials` prop (API truth) enters CollectPaymentPanel but is **never used** in the bill→payment→API path. It only appears in room transfer display logic.

**OPERATIONAL IMPLICATION**: Any change to `CollectPaymentPanel.getItemLinePrice()` or `taxTotals` directly changes what the backend receives as payment amount. Any change to `buildCartItem()` does NOT affect collect-bill (different code path). These two paths are **decoupled** — there is no shared calculation function between them.

See **Section 8** for the full refined financial framework.

### RULE-2: Socket vs API Conflict Resolution

**The GET API response is ALWAYS the final authority. Socket data is a trigger, not truth.**

This is already the codebase's design:
- `update-order-status` ignores socket status, fetches API — `socketHandlers.js:290`
- `new-order` uses socket payload temporarily, then overwrites with GET response — `socketHandlers.js:165-180`
- All other events fetch from API and write to context

**OPERATIONAL IMPLICATION**: If socket delivers data that contradicts a subsequent GET call, the GET call wins. Never branch on socket-provided status values in new code.

### RULE-3: Table Engage/Release Interpretation

**Until BC-2 is confirmed, maintain the BUG-216 workaround: treat `free` as `engage`.**

The reasoning:
- If `free` truly means free, locking for ~1-2s (until GET enrichment releases) is a minor UX delay
- If `free` arrives during cancel-item and we DON'T lock, the table becomes clickable while its data is stale, leading to incorrect order display

**Release timing**: Lock is ALWAYS released after GET enrichment via double `requestAnimationFrame`. This is the ONLY unlock path for order-triggered events.

**OPERATIONAL IMPLICATION**: If adding new socket handlers that affect table state, always include the double-rAF unlock pattern. Never call `setTableEngaged(false)` synchronously.

**OPERATIONAL IMPLICATION**:

## 5. Critical Contradictions

Each contradiction below represents a real code conflict, not a hypothetical concern. All verified line-by-line.

### CONTRADICTION-1: Three Divergent Tax Calculations

**What**: Tax on the same items is calculated differently in 3 places.

**Location A — `buildCartItem()` (`orderTransform.js:262-278`)**
```
lineTotal = (basePrice + addonAmount + variationAmount) * qty
taxAmount = isInclusive ? lineTotal - (lineTotal / (1 + pct/100)) : lineTotal * (pct/100)
```
- Tax base: `fullUnitPrice * qty` where `fullUnitPrice = basePrice + addonTotal + variationTotal`
- Addons/variations ARE included in tax base

**Location B — `OrderEntry.jsx` (lines 335-341)**
```
linePrice = item.totalPrice || (item.price * item.qty)
taxAmount = isInclusive ? linePrice - (linePrice / (1 + pct/100)) : linePrice * (pct/100)
```
- Tax base: `item.price * qty` (or `item.totalPrice` if set)
- `item.price` is the bare base price for unplaced items (no addons/variations)
- Addons/variations are **NOT** included in tax base for unplaced items

**Location C — `CollectPaymentPanel.jsx` (lines 48-86)**
```
linePrice = getItemLinePrice(item)  // = base*qty + (addonSum + varSum)*qty
taxAmount = isInclusive ? linePrice - (linePrice / (1 + pct/100)) : linePrice * (pct/100)
```
- Tax base: `base*qty + (addonSum + varSum)*qty`
- Addons/variations ARE included in tax base
- BUT: addon calculation formula differs from buildCartItem:
  - `buildCartItem`: `addonAmount = sum(addon.price * addon.qty)` per addon, then `fullUnitPrice = base + addonAmount + varAmount` (PER UNIT), then `* itemQty`
  - `getItemLinePrice`: `addonSum = sum(addon.price * addon.qty)` then `(addonSum + varSum) * item.qty` — multiplies raw addon sum by item qty again

**Concrete divergence scenario**:
Item: price=100, 1 addon at price=20 qty=2, item qty=3, tax 5% Exclusive

- **buildCartItem**: fullUnitPrice = 100 + 40 = 140; lineTotal = 140*3 = 420; tax = 420*0.05 = 21
- **OrderEntry**: linePrice = 100*3 = 300; tax = 300*0.05 = 15 ← **DIFFERENT (misses addons)**
- **getItemLinePrice**: addonSum=40; base=300; linePrice = 300 + 40*3 = 420; tax = 420*0.05 = 21 ← **matches buildCartItem by coincidence here, but addon double-count risk exists with per-addon qty > 1 and item qty > 1**

**Status**: CONFIRMED divergence. Location B (OrderEntry cart total) will show a lower tax than Location A (API payload) and Location C (payment panel) for any item with addons or variations.

### CONTRADICTION-2: CollectPaymentPanel Sends Locally Calculated Totals, Not API-Provided Totals

**What**: For existing orders, the payment panel recalculates `itemTotal`, `sgst`, `cgst` from cart items locally, ignoring the `orderFinancials` from the API (socket-enriched GET response).

**Traced data flow for Collect Bill on existing order**:

2. `CollectPaymentPanel.jsx:64-86` — `sgst`, `cgst` ← LOCAL CALC
3. `CollectPaymentPanel.jsx:195-213` — `paymentData = { finalTotal, sgst, cgst, itemTotal }` ← LOCAL VALUES
4. `OrderEntry.jsx:731` — `orderToAPI.collectBillExisting(effectiveTable, cartItems, customer, paymentData)` ← LOCAL VALUES PASSED
5. `orderTransform.js:595-640` — payload built from `paymentData.finalTotal`, `paymentData.sgst + paymentData.cgst`, `paymentData.itemTotal` ← LOCAL VALUES SENT TO API

**The `orderFinancials` prop** (which holds API-provided `amount`, `subtotalAmount`) **is received by CollectPaymentPanel but never used for the actual bill calculation or payment payload.** It's only used for display in room transfer flows.

**Impact**: The amount sent to the backend for collect-bill may differ from what the backend calculated when the order was placed/updated. This is a **financial discrepancy path**.

---

### CONTRADICTION-3: `orderItemsByTableId` (last-write) vs `getOrderByTableId` (first-match)

**What**: Two methods in OrderContext for finding an order by tableId use opposite strategies.

- `getOrderByTableId(tableId)` → `orders.find(o => o.tableId === tableId)` → returns FIRST match — `OrderContext.jsx:157-159`
- `orderItemsByTableId` useMemo → `for (const order of orders) { map[order.tableId] = {...} }` → keeps LAST match — `OrderContext.jsx:192-216`

If `orders` array contains `[orderA(table=5), orderB(table=5)]`:
- `getOrderByTableId(5)` returns `orderA`
- `orderItemsByTableId[5]` returns `orderB`

**Impact**: DashboardPage uses `getOrderByTableId` in `useMemo` (line 187) for table enrichment, but uses `orderItemsByTableId` for order data in `getOrderDataForEntry` (line 474). These could return different orders for the same table.

---

### CONTRADICTION-4: Round-Off Sign Convention

**What**: `calcOrderTotals` produces `round_up` as a positive number (or 0). `CollectPaymentPanel` produces `roundOff` which can be negative.

- `calcOrderTotals`: `roundUp = diff >= 0.10 ? diff : 0` → always ≥ 0 — `orderTransform.js:328`
- `CollectPaymentPanel`: `roundOff = roundDiff >= 0.10 ? roundDiff : -Math.round(...)` → can be negative — `CollectPaymentPanel.jsx:157`

Both arrive at the same `finalTotal`/`orderAmount` (ceil or floor), but the `roundOff`/`round_up` value sent to the backend differs in sign for the floor case:
- `calcOrderTotals`: sends `round_up: "0.00"` for floor case
- `collectBillExisting`: sends `round_up: 0` (hardcoded, doesn't use CollectPaymentPanel's roundOff)

**Impact**: Low — the `round_up` field in `collectBillExisting` is hardcoded to 0 regardless (line 617), so the display discrepancy doesn't affect the API. But if this field is ever wired, the sign mismatch will cause issues.

---

### CONTRADICTION-5: `update-food-status` Has No Engage Lock

**What**: `handleUpdateFoodStatus` calls `fetchOrderWithRetry → updateOrder → syncTableStatus` but does NOT release the engaged lock. Neither does it acquire one.

Compare:
- `handleUpdateOrder`: releases lock via `setTableEngaged(false)` after GET — `socketHandlers.js:226-232`
- `handleUpdateFoodStatus`: no lock interaction at all — `socketHandlers.js:244-269`

**Impact**: If `update-table engage` fires (from backend) before `update-food-status`, the table gets locked. `update-food-status` handler completes but doesn't unlock. The table stays locked until another event (update-order, update-order-status) releases it.

This creates an asymmetry: the same table operation can leave a table locked or unlocked depending on which socket events the backend sends and in what order.

## 6. System Invariants

Non-negotiable truths observed in the codebase. These hold regardless of backend behavior, configuration, or future feature development. Violating any of these during development will break the system.

---

### INV-1: Order Lifecycle Rules

**INV-1a: Orders are identified by `orderId` (numeric). This is globally unique across the system.**
- `addOrder` deduplicates by `orderId` — `OrderContext.jsx:63`
- `updateOrder` matches by `orderId` — `OrderContext.jsx:89`
- `removeOrder` filters by `orderId` (coerced to Number) — `OrderContext.jsx:114,118`
- All socket handlers extract `orderId` from message index 1 — `socketHandlers.js:48`

**INV-1b: An order exists in OrderContext from the moment `addOrder()` is called until `removeOrder()` is called. There is no other lifecycle.**
- No TTL, no expiry, no garbage collection
- `addOrder` can be called multiple times for the same orderId — it upserts (update-if-exists) — `OrderContext.jsx:64-67`
- `updateOrder` for a non-existent orderId will INSERT it — `OrderContext.jsx:91-93`
- `removeOrder` for a non-existent orderId is a silent no-op (filter returns same array)

**INV-1c: The ONLY code path that removes an order from context is `removeOrder()`, and it is ONLY called from `handleUpdateOrderStatus` in `socketHandlers.js:300,309`.**
- `handleCancelOrder` in OrderEntry does NOT call removeOrder directly — it calls the cancel API and then polls `waitForOrderRemoval`, which waits for the socket handler to do the removal.
- No timer, no sweep, no UI action directly removes orders.

**INV-1d: An order's `tableId` field is immutable after initial creation from the API response.**
- `fromAPI.order()` sets `tableId: api.table_id || 0` — `orderTransform.js:124`
- No code path modifies `tableId` after the order object is created
- Shift-table and merge-table operations work by calling APIs that create new order records server-side; the frontend reacts to socket events for the new state

**INV-1e: Walk-in orders are distinguished by `tableId === 0` or `isWalkIn === true`.**
- `isWalkIn = !api.table_id || api.table_id === 0` — `orderTransform.js:104`
- Both `getOrderByTableId` and `orderItemsByTableId` exclude walk-ins — `OrderContext.jsx:158,196`
- Walk-ins appear as virtual entries on the dashboard with `id: 'wc-${orderId}'` — `DashboardPage.jsx:225-238`

---

### INV-2: API vs Socket Authority

**INV-2a: The REST API is the sole authority for order data. Sockets are triggers, not sources.**
- 6 of 7 socket event handlers call `fetchOrderWithRetry()` to get the real data — `socketHandlers.js:89-109`
- The one exception (`new-order`) uses inline payload TEMPORARILY, then overwrites with GET API data — `socketHandlers.js:165-180`
- `update-order-status` explicitly ignores socket's status field — `socketHandlers.js:290`

**INV-2b: The REST API is the sole authority for table data at load time. After load, table status is derived from TWO sources:**
1. Socket `update-table` events → `updateTableStatus()` or `setTableEngaged()` — `socketHandlers.js:381-414`
2. Socket order events → `syncTableStatus()` derives table status from order data — `socketHandlers.js:123-132`

There is no periodic re-fetch of table data during a session (only manual refresh via `useRefreshAllData`).

**INV-2c: `updateTableStatus` never creates tables. It can only modify existing entries.**
- If `tableId` not found in `tables[]`, it logs a warning and returns unchanged state — `TableContext.jsx:107-111`
- Tables are only added during `setTables()` (LoadingPage) or `refreshTables()` (manual refresh)

---

### INV-3: Order Removal Behavior

**INV-3a: Order removal and table status clearing are two separate operations that happen to coincide.**
- `handleUpdateOrderStatus` calls `syncTableStatus(order, updateTableStatus)` FIRST, then `removeOrder(orderId)` — `socketHandlers.js:299-300`
- `syncTableStatus` calls `updateTableStatus(tableId, status)` which, if status is `'available'`, clears order display fields on the table — `TableContext.jsx:122-130`
- These are two independent context mutations. There is no transaction. A React render between them would show the table as "available" with the order still in OrderContext.

**INV-3b: `removeOrder` normalizes orderId to Number before filtering.**
- Socket sends orderId as string, context stores as number — `OrderContext.jsx:114`
- This normalization is the ONLY type coercion in the removal path

**INV-3c: Removal of a non-existent orderId is safe (no-op).**
- `prev.filter(o => o.orderId !== numericId)` returns the same array if no match — `OrderContext.jsx:118`

---

### INV-4: State Ownership Boundaries

**INV-4a: Context state is NEVER directly mutated. All mutations go through setter functions exposed via context value.**
- `setOrdersState(prev => ...)` pattern used consistently — verified in OrderContext, TableContext
- Socket handlers receive setter references via `actionsRef.current` — `useSocketEvents.js:44`
- No `orders.push()`, no `tables[i].status = ...` anywhere in codebase

**INV-4b: Each context is independent. No context reads from another context's internal state.**
- The provider tree (Auth → Socket → Restaurant → Menu → Table → Settings → Order) establishes dependency ORDER, but no context imports another
- The only place multiple contexts interact is in consumer components (DashboardPage, OrderEntry, socket handlers)

**INV-4c: `ordersRef` and `engagedTablesRef` exist ONLY for synchronous polling. They are NOT a second source of truth.**
- Both refs are updated in the same `setXState(prev => { ...; ref.current = next; return next; })` pattern
- `ordersRef` is read by `waitForOrderRemoval` — `OrderContext.jsx:169`
- `engagedTablesRef` is read by `waitForTableEngaged` — `TableContext.jsx:76`
- Neither ref is read for rendering. Only state drives renders.

---

### INV-5: Dashboard Consistency Model

**INV-5a: The dashboard is an eventually-consistent derived view. It is NOT transactionally consistent.**

The dashboard grid is built by `useMemo` that joins:
- `apiTables` from TableContext (physical tables)
- `getOrderByTableId(t.tableId)` from OrderContext (running orders)
- `walkInOrders` from OrderContext (virtual walk-in entries)

These inputs are updated independently by socket handlers. Between updates, the dashboard may show:
- A table as "available" while its order still exists in OrderContext (table status updated before order removed)
- An order in the OrderContext grid without a matching table status update (order added before table status synced)

**This is by design.** The `useMemo` recalculates on every state change, and the engaged lock prevents user interaction during the transient inconsistency window.

**INV-5b: The dashboard re-derives on EVERY change to `tablesLoaded`, `apiTables`, `getOrderByTableId`, or `walkInOrders`.**
- `DashboardPage.jsx:268` — useMemo dependency array: `[tablesLoaded, apiTables, getOrderByTableId, walkInOrders]`
- No caching, no staleness check. Every context mutation triggers a full re-derive.

**INV-5c: DashboardPage uses `getOrderByTableId` (FIRST match) for grid enrichment, but `orderItemsByTableId` (LAST match) for `getOrderDataForEntry` (opening OrderEntry).**
- Grid enrichment: `DashboardPage.jsx:187` → `getOrderByTableId(t.tableId)` → `OrderContext.jsx:158` → `orders.find()`
- Order data for OrderEntry: `DashboardPage.jsx:474` → `orderItemsByTableId[tableEntry.tableId]` → `OrderContext.jsx:197` → last-write-wins map
- If RV-4 confirms multiple orders per table are impossible, both return the same result. If multiple exist, they diverge (CONTRADICTION-3).

## 7. Clarified Definitions

Precise, operational definitions for terms used throughout this document and the codebase. These are NOT business definitions — they describe what the code actually does.

---

### DEF-1: "Engaged"

**What it IS**: A frontend-only, per-table UI interaction lock stored in a `Set<tableId>` within TableContext.

**What it is NOT**:
- NOT a business state (there is no "engaged" status in the backend)
- NOT a persistent state (lost on page refresh)
- NOT a server-communicated state (the backend sends `engage`/`free` socket events, but the frontend lock is managed independently)

**Precise behavior**:
- When a table is engaged, `DashboardPage.handleTableClick` returns early without opening OrderEntry — `DashboardPage.jsx:547-549`
- The table card may show a visual indicator (opacity/spinner) via `isEngaged` prop — `DashboardPage.jsx:722`
- Engaged does NOT prevent API calls. It does NOT prevent socket updates. It ONLY prevents user click-through on the dashboard.

**Lifecycle**:
```
LOCKED by: socket 'engage' event | socket 'free' event (BUG-216) | new-order handler | collectBill in OrderEntry
UNLOCKED by: socket handler after GET API enrichment + double requestAnimationFrame
TIMEOUT: waitForTableEngaged resolves after 5000ms; no automatic unlock on timeout
RESET: Full page refresh clears all engaged state (Set is in memory)
```

**The "double requestAnimationFrame" pattern explained**:
```js
requestAnimationFrame(() => {        // ← React has committed DOM updates
  requestAnimationFrame(() => {      // ← Browser has painted those updates
    setTableEngaged(tableId, false); // ← NOW safe to unlock: user sees fresh data
  });
});
```
Purpose: Ensures the user cannot click a table between "context updated" and "screen painted." Without this, a fast click could open OrderEntry with the pre-update data still visually displayed.

---

### DEF-2: "Source of Truth" for Financials

The system has NO single source of truth for financial values. Instead, it has a **pipeline of calculations** where different stages serve different purposes:

```
Stage 1: PREVIEW (display-only, never sent to API)
  ├── OrderEntry.jsx cart total (lines 331-364)
  │   Purpose: Show approximate total to user before placing order
  │   Includes: base price × qty + tax (EXCLUDING addon/variation — CONTRADICTION-1)
  │   Used for: "Collect Bill" button label, cart footer display
  │
Stage 2: AUTHORITATIVE SUBMISSION (sent to API, assumed trusted by backend)
  ├── buildCartItem() + calcOrderTotals() — for Place Order & Update Order
  │   Purpose: Compute exact financial payload for backend
  │   Includes: base + addons + variations, per-item tax, order-level rounding
  │   Used for: POST/PUT to place-order, update-place-order endpoints
  │
  ├── CollectPaymentPanel local calc — for Collect Bill
  │   Purpose: Compute bill summary AND payment payload
  │   Includes: base + addons + variations (via getItemLinePrice), per-item tax, discounts, round-off
  │   Used for: POST to order-bill-payment endpoint
  │   WARNING: This is a SEPARATE code path from buildCartItem. They are not shared.
  │
Stage 3: API ECHO (backend's response, read-only on frontend)
  ├── orderFinancials in OrderEntry (from GET single-order via socket handler)
  │   Purpose: Display backend-calculated totals after order is placed
  │   Includes: amount, subtotalAmount, subtotalBeforeTax — as returned by API
  │   Used for: Dashboard amount display, OrderEntry cart total (post-place)
  │   NOT used for: Collect Bill payload (CONTRADICTION-2)
```

**Key insight**: Stage 2 has TWO independent calculators (buildCartItem vs CollectPaymentPanel) that take different code paths for the same items. Stage 3 (the backend's own answer) is available but NOT fed back into Stage 2 for collect-bill.

---

### DEF-3: "Socket Event" vs "API Enrichment"

These are two distinct phases of a single real-time update cycle:

**Socket Event** = The raw message received on a Socket.IO channel.
- Shape: `[eventName, orderId, restaurantId, status, payload?]` — an array, not an object
- Content: Minimal. Only `new-order` includes full order data. All others are just an orderId trigger.
- Latency: Near-instant (WebSocket push)
- Trust: LOW — the codebase explicitly ignores socket-provided status values (BUG-217 workaround)

**API Enrichment** = The GET request triggered by a socket event to fetch authoritative data.
- Endpoint: `POST /api/v2/vendoremployee/get-single-order-new` with `{ order_id }` body
- Content: Full order data (51+ fields) including financial totals, item details, table info
- Latency: ~200-1000ms (HTTP round-trip)
- Trust: HIGH — this is what gets written to OrderContext
- Retry: 1 retry with 1-second delay — `socketHandlers.js:85-109`

**The complete update cycle**:
```
Socket Event arrives
  → Handler parses message (extract orderId)
  → [For new-order only]: Transform inline payload → addOrder() (temporary)
  → ASYNC: fetchOrderWithRetry(orderId) — GET API call
  → Transform API response via fromAPI.order()
  → updateOrder() or removeOrder() in OrderContext
  → syncTableStatus() in TableContext
  → setTableEngaged(false) via double rAF (if applicable)
```

**The gap between socket event and API enrichment is the "engaged" window.** The table is locked to prevent user interaction with stale data.

## 8. Refined Financial Rule

This section replaces the "OPERATIONAL IMPLICATION" paragraph of RULE-1 with a complete decision framework for financial calculations.

---

### 8.1 The Two Financial Pipelines

The codebase has two completely independent financial calculation pipelines. They share NO code.

**Pipeline A: Order Placement** (Place New Order + Update Existing Order)
```
Cart items (from OrderEntry local state)
  → buildCartItem() per item                    [orderTransform.js:214-301]
    → calculates: basePrice, addonAmount, variationAmount, fullUnitPrice
    → calculates: per-item gst_amount, vat_amount
    → outputs: { food_id, quantity, price, gst_amount, vat_amount, _fullUnitPrice, ... }
  → calcOrderTotals() over all built items       [orderTransform.js:309-339]
    → sums: subtotal, gstTax, vatTax
    → calculates: rounding (ceil if diff >= 0.10, else floor)
    → outputs: { order_sub_total_amount, tax_amount, gst_tax, vat_tax, order_amount, round_up }
  → Sent to API as part of placeOrder/updateOrder payload
```

**Pipeline B: Collect Bill** (Payment on Existing Order)
```
Cart items (placed items synced from OrderContext + any unplaced)
  → getItemLinePrice() per item                  [CollectPaymentPanel.jsx:48-59]
    → calculates: base*qty + (addonSum + varSum)*qty
  → taxTotals useMemo                            [CollectPaymentPanel.jsx:64-86]
    → calculates: per-item tax, split into sgst/cgst for GST type
  → discount, coupon, loyalty, wallet deductions  [CollectPaymentPanel.jsx:127-145]
  → round-off                                    [CollectPaymentPanel.jsx:154-158]
  → handlePayment() builds paymentData            [CollectPaymentPanel.jsx:192-213]
  → OrderEntry.onPaymentComplete() calls collectBillExisting(table, cart, customer, paymentData)
  → collectBillExisting() builds API payload      [orderTransform.js:595-640]
  → Sent to API as POST to order-bill-payment
```

### 8.2 Where They Diverge

| Aspect | Pipeline A (buildCartItem) | Pipeline B (CollectPaymentPanel) |
|--------|---------------------------|----------------------------------|
| Addon price model | `addonAmount = sum(addon.price * addon.qty)` then `fullUnitPrice = base + addonAmount + varAmount` PER UNIT, then `× itemQty` | `addonSum = sum(addon.price * addon.qty)` then `(addonSum + varSum) × itemQty` added to `base × itemQty` |
| Variation price model | Handles `selectedVariants` (object) AND `variation` (array) with two branch paths | Only handles `variation` (array format from placed items) |
| Tax scope | ALL items including unplaced (for place/update) | ALL active (non-cancelled) items including placed + unplaced |
| Discount support | None — `discount_amount: '0.00'` hardcoded per item | Full: manual, preset, coupon, loyalty, wallet |
| Round-off sent to API | `round_up: String(roundUp.toFixed(2))` — positive or "0.00" | `round_up: 0` — hardcoded in collectBillExisting, not from CollectPaymentPanel's roundOff |
| GST field names | `gst_tax`, `vat_tax` | `total_gst_tax_amount` AND `gst_tax` (both sent) |

### 8.3 Safe Development Rules

**RULE-8a**: Never assume Pipeline A and Pipeline B produce the same result for the same items. They don't, and they can't without shared code.

**RULE-8b**: When modifying financial logic, identify WHICH pipeline you're in:
- Changing `buildCartItem()` or `calcOrderTotals()` → affects Place Order and Update Order ONLY
- Changing `getItemLinePrice()` or `taxTotals` in CollectPaymentPanel → affects Collect Bill ONLY
- Changing `OrderEntry.jsx` local tax calculation → affects DISPLAY ONLY (preview)

**RULE-8c**: After any financial logic change, test ALL three scenarios independently:
1. Place new order → verify `order_amount` in API request matches expected
2. Update existing order → verify combined totals in `cart-update` payload
3. Collect bill on existing order → verify `payment_amount` in API request matches expected

**RULE-8d**: Until CONTRADICTION-2 is resolved, the Collect Bill flow must be tested by comparing:
- The `payment_amount` sent to API (from Pipeline B)
- The `order_amount` that was sent when the order was originally placed (from Pipeline A)
- The `amount` field in the GET single-order response (backend's truth)
If all three disagree, the system has a financial integrity issue.

**RULE-8e**: `orderFinancials` (API echo) should be treated as the DISPLAY truth for existing orders. If a future refactor wires CollectPaymentPanel to use `orderFinancials.amount` instead of local recalculation, Pipeline B's divergence problem (CONTRADICTION-2) would be resolved. This is the recommended direction but requires confirming that `orderFinancials` includes discount/tip/service-charge adjustments the payment panel currently computes locally.
